# AI 时代程序设计原则

Part A：设计原则（7 条）— 代码本身应该长什么样
Part B：AI 协作工作流（3 条）— 怎么和 AI 协作才能落实这些原则

推导依据：
- 正向：AI 让重写变便宜、让生成变快、让上下文成为瓶颈
- 反向：AI 代码的实证缺陷（逻辑错误 1.7x、安全漏洞 2.74x、happy path 偏见、职责混乱）

---

# Part A：设计原则

## 原则 1：规格先行，测试落地（Spec First, Tests Enforce）

**核心：** 先写清楚"做什么"和"怎么算对"，再把验收标准翻译成测试，最后写实现。

**为什么 AI 时代更重要：** AI 没有规格就会猜。AI 自己写的测试往往在测自己的假设。打断自我验证循环：你定义验收标准 → 转化为测试 → AI 写实现来通过测试。

**实践：**
- 每个功能先写 Given-When-Then 验收标准
- 验收标准 → 集成测试，边界情况 → 单元测试
- 接口用类型定义约束
- 需求变更：改规格 → 改测试 → 改代码，顺序不能反
- 测试只测行为和输出，不测实现细节

**CLAUDE.md 规则示例：**
```
- 新功能必须先有验收标准再写代码
- 先写测试再写实现。测试描述用行为语言：should [行为] when [条件]
- 接口变更必须先更新类型定义
- 修改现有功能前，先跑现有测试确认基线
```

---

## 原则 2：显式约束（Explicit Constraints）

**核心：** 所有约定写出来，所有禁止项列出来。给 AI 越多约束，输出越可控。

**为什么 AI 时代更重要：** AI 不加约束时选"最常见"模式而非"最合适"模式。隐式约定是 AI 最大的错误来源。

**实践：**
- 技术栈锁定，不让 AI 自行选择
- 优先使用成熟开源库：能用现成库解决的问题不自己写。AI 特别爱从零实现已有轮子——明确告诉它用什么库。
- 禁止清单比正面指导更有效
- 函数名体现副作用，配置外置不硬编码
- 文件 < 200 行，函数 < 30 行
- 错误处理显式写出

**CLAUDE.md 规则示例：**
```
## 技术栈（锁定，不要自行更换）
[在此列出具体技术栈和版本]

## 命名规范
- 函数命名必须体现副作用

## 尺寸限制
- 单文件不超过 200 行
- 单函数不超过 30 行

## 禁止项
- ❌ [列出项目特定的禁止模式]
- ❌ 硬编码配置值
- ❌ 依赖全局错误捕获

## UI 规范（如有前端，参考 ux-principles skill）
- [指定组件库、颜色方案、间距系统]
```

---

## 原则 3：为可删除性设计（Design for Deletability）

**核心：** 写容易删除的代码，不是容易扩展的。AI 让重写成本断崖下降，可删除性比可扩展性更有价值。

**实践：**
- 模块间通过类型接口通信，不导入内部文件
- 每个模块自包含：类型、测试、单一入口
- 避免跨模块共享可变状态
- 第三方依赖封装一层
- 判断标准：删掉整个目录，只改接口层系统能继续跑？

**CLAUDE.md 规则示例：**
```
- 模块间通过公共接口/类型通信，禁止直接导入其他模块的内部文件
- 功能模块自包含在独立目录下
- 第三方库通过封装层使用
```

---

## 原则 4：数据模型优先（Data Model First）

**核心：** 在**实现阶段**，先建数据模型（schema、迁移），再写业务逻辑，最后做界面。

> **注意区分两个阶段的顺序：**
> - 需求阶段：先梳理用户流程 → 从流程推导数据模型 → 再定义界面（流程驱动）
> - 实现阶段：先建数据模型 → 再写业务逻辑 → 最后做界面（模型驱动）

**实践：**
- 规格文档中定义字段、类型、约束、关系
- 约束在数据库层强制（NOT NULL、UNIQUE、FK）
- 用迁移工具管理 schema 演变
- 新功能从"需要什么数据"开始实现

---

## 原则 5：边界清晰的分层（Clean Boundaries）

**核心：** 每层只做自己的事，不穿透不逆向。

```
入口层（路由/CLI/事件处理） → 只做输入解析和输出格式化
        ↓
服务层（业务逻辑） → 编排操作，包含核心规则。这是系统中唯一"聪明"的层。
        ↓
数据层（存储访问） → 封装所有存储操作。保持"笨"——只做 CRUD，不含业务判断。
        ↓
外部层（第三方集成） → 封装外部 API。保持"笨"——只做调用和格式转换，不做决策。
```

> **Smart Agent, Dumb Tools 思想：** 复杂的判断、编排、业务规则集中在服务层——这是 AI 最能理解和推理的地方。数据层和外部层保持简单确定：输入什么就输出什么，没有隐藏行为。不要在数据库触发器、ORM 钩子、中间件、消息队列消费者里藏业务逻辑——它们是"工具"，应该保持"笨"。

**CLAUDE.md 规则示例：**
```
- 入口层只做输入解析和输出格式化，业务逻辑放服务层
- 服务层不直接操作存储，通过数据层访问
- 外部 API 调用封装在独立模块中
- 禁止跨层传递存储层内部对象
- 禁止在数据库触发器、ORM 钩子、中间件中编写业务逻辑
```

---

## 原则 6：为失败而设计（Design for Failure）

**核心：** 假设一切都会出错。AI 有严重的 happy path 偏见——安全漏洞率比人类高 2.74 倍。

**实践：**
- 每个外部调用处理失败 + 超时
- 区分可重试/不可重试错误
- 用户输入永远不可信
- 统一错误响应格式
- 每个 catch 块做有意义的事，不允许空 catch

**CLAUDE.md 规则示例：**
```
- 所有外部调用必须有错误处理和超时
- 用户输入在入口层验证和清洗
- 使用统一的错误响应格式
- 异步操作必须有失败处理路径
- 每个 catch/except 块必须做有意义的事，不允许空 catch
```

---

## 原则 7：管理演进（Manage Evolution）

**核心：** AI 不会主动提议重构。你必须识别腐化信号并干预。

**重构信号：** 同一逻辑 >3 处、文件 >300 行、改一个功能动 >4 文件、AI 新旧代码不一致、测试脆弱、新对话看不懂代码意图。

**实践：**
- 每 3-5 切片后做技术债审查
- ADR 记录架构决策
- 临时方案标注 TODO 和预期替换时间
- 重构时用工作流 W2 的方式——小步改、每步有测试保护
- 定期用 `change-guard` 工作流 4（文档同步）检查文档是否需要更新 ADR
- 当技术债积累到影响迭代速度时，在 `product-iteration` 的迭代决策中将重构列为候选项，用 ICE 评分和其他需求一起排优先级——不要单独找时间重构，而是让它和业务需求竞争资源

# Part B：AI 协作工作流

**元原则：约束是有成本的**

如果移除某个约束后模型表现不变或更好，就删掉它。每次模型大版本升级后，用 skill-admin 的退役流程审查现有 Skill 和 CLAUDE.md 规则。

---

## 工作流 1：上下文工程（Context Engineering）

**核心：** AI 输出质量由输入上下文质量决定。用最少的高信号 token 最大化结果。

**上下文层级：**
```
全局（~/.claude/CLAUDE.md）→ 通用偏好和规则
项目（./CLAUDE.md）→ 技术栈、目录结构、命令，< 150 行
目录（子目录/CLAUDE.md）→ 模块级规则（如 frontend/ 和 backend/ 用不同规范），大型项目按需使用
任务（每次 prompt）→ 具体规格和文件引用
领域（docs/ 按需引入）→ 架构决策、业务规则
```

渐进式披露：只给当前任务需要的上下文，不全塞进去。

冲突解决：下层覆盖上层（目录级 > 项目级 > 全局级）。任务级 prompt 中的显式指令优先于所有文件级规则。

**长任务上下文策略：**

不同任务需要不同的上下文管理策略。选择依据是**信息丢失的代价**：

| 判断信号 | 策略 | 做法 |
|---------|------|------|
| Task 已完成，准备切换到下一个 | /clear | 清空上下文，干净开始。最简单，丢失最多 |
| 对话变长，模型开始遗漏早期讨论或重复已有信息 | 主动压缩 | 说"总结当前进展和待解决问题"刷新关注重点；若质量仍下降，/clear 后带总结开新对话 |
| 任务跨多个 session，或项目持续数天以上 | Structured Notes | 维护 `docs/progress.md`，每次 session 结束更新、开始时先读 |
| 多个独立子任务可同时进行 | Sub-Agent | 派 subagent 并行，只回收摘要到主 context |

> 参考值：单 Task 通常 10 轮内；超过 15 轮大概率需要主动压缩。以实际质量下降信号为准。

**高级模式：代码执行替代工具调用**

MCP 工具 > 10 且工具间有数据流转时，将工具组织为文件系统结构，agent 通过写代码调用而非直接 tool call。工具定义按需加载，中间数据在执行环境内流转不进出 context。前提是有安全的代码执行沙箱。

---

## 工作流 2：可验证的增量交付（Verifiable Increments）

**核心：** 小块做完验证，对了固化，错了回滚。

**执行节奏：**
```
1. 变更前：用 change-guard 做影响分析，确认爆炸半径
2. AI 写测试（基于验收标准）
3. AI 写实现（让测试通过）
4. 变更后：用 change-guard 做完整性检查
5. 跑全量测试
6. git commit
7. /clear（清理 AI 上下文）
8. 下一个任务
```

**关键原则：**
- 每个增量是垂直切片——从数据层到入口层完整贯通，不是只做一层
- 每个切片 < 10 轮对话能完成，超过说明任务太大需要再拆
- Git 是安全网——每个切片一次 commit，出了问题可以精确回滚
- /clear 是卫生习惯——切换任务前清理上下文，避免 AI 被前一个任务的记忆污染

> 任务的具体格式和拆分方法见 `product-builder` 阶段 4 的任务模板。

**并行执行（当 docs/tasks.md 中有标注"可并行"的 Task 时）：**

三种方式，按复杂度递增：

1. **Subagent（最轻量，推荐起步）：** 在当前会话中让 Claude 用 Task tool 派出子任务。适合 2-3 个独立的小任务。
   ```
   请用 subagent 并行完成以下任务：
   - Task 2.1：[描述]
   - Task 2.2：[描述]
   每个 subagent 完成后汇报结果，我来 review。
   ```

2. **Agent Teams（中等，适合 3-5 个独立任务）：** Claude 自动协调多个会话，队友之间可以通信。
   ```
   请用 Agent Teams 并行执行 Task 2.1-2.4：
   - 每个 teammate 负责一个 Task
   - 完成后统一汇报，我来 review 和合并
   ```
   前提：设置 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`

3. **Git Worktree + 手动多会话（最灵活，适合大型并行）：** 每个 Task 在独立的 worktree 中执行，避免文件冲突。
   ```
   # 创建独立工作区
   git worktree add ../task-2.1 -b feat/task-2.1
   git worktree add ../task-2.2 -b feat/task-2.2

   # 在不同终端分别启动 Claude Code
   cd ../task-2.1 && claude
   cd ../task-2.2 && claude

   # 各自完成后合并回主分支
   git checkout main
   git merge feat/task-2.1
   git merge feat/task-2.2

   # 清理
   git worktree remove ../task-2.1
   git worktree remove ../task-2.2
   ```

**并行执行的前提条件：**
- 并行的 Task 之间不能修改同一个文件
- 并行的 Task 之间没有数据依赖
- 每个 Task 的验收标准是独立可验证的
- 并行完成后，合并前跑全量测试确认没有冲突

**并行完成后的合并流程：**
```
1. 所有并行 Task 完成
2. 逐个合并到主分支（如果有冲突，手动解决）
3. 合并后跑全量测试
4. 用 change-guard 工作流 2 做完整性检查
5. 一次性 commit 合并结果
```

> 如果安装了 claude-workflow 插件，以上并行流程可以用 `/cw-dispatch`（自动派发 subagent）和 `/cw-worktree`（自动管理 worktree 生命周期）来替代手动操作。

## 工作流 3：审查与学习循环（Review & Learn Loop）

**核心：** AI 生成的每一行代码都需要人类审查。审查不只是找 bug——还要把 AI 错误模式沉淀为规则。

**Writer/Reviewer 分离原则：**
同一个 Claude Code 会话对自己刚写的代码有认知偏见。用一个**新的会话**来审查可以打破这个偏见：
```
# 会话 A：写代码
> 完成 Task 2.1，commit

# 会话 B：审查（新终端或 /clear 后）
> 请 review 最近一次 commit 的 diff，重点关注安全、边界情况和错误处理
```

**审查三问（快速自检）：**
1. **边界覆盖了吗？** 空值、极端值、恶意输入、并发情况
2. **失败处理了吗？** 网络断开、服务不可用、数据不存在、权限不足
3. **做了我没要求的事吗？** AI 爱"加戏"——添加未要求的功能、引入新依赖、做过度优化

> 完整的检查清单和高频 bug 模式表见 `change-guard` 工作流 2 和工作流 3。
> 这里的三问是快速自检，change-guard 的清单是系统检查——两者粒度不同，不重复。

**学习循环：**
```
AI 犯错 → 分析错误模式 → 加入 CLAUDE.md 禁止项 → AI 下次不再犯
```

每次 AI 犯了有代表性的错误，把它抽象成一条规则写进 CLAUDE.md。几周下来，你的 CLAUDE.md 会变成一份精准的"AI 使用手册"——它是从真实错误中生长出来的，比任何通用模板都有效。

## Part A：设计原则

| # | 原则 | 一句话 | 核心规则 |
|---|------|--------|---------|
| 1 | 规格先行，测试落地 | 规格→测试→代码，不能反 | 先有验收标准，先写测试 |
| 2 | 显式约束 | 写出来、列出来、禁出来 | 锁定技术栈，明确禁止项 |
| 3 | 为可删除性设计 | 模块能整体替换不影响系统 | 接口通信，自包含模块 |
| 4 | 数据模型优先 | 先设计数据，再写逻辑和界面 | 新功能先定义数据模型 |
| 5 | 边界清晰的分层 | 入口→服务→数据→外部，不穿透 | 每层只调下层 |
| 6 | 为失败而设计 | 假设一切都会出错 | 外部调用必须有错误处理和超时 |
| 7 | 管理演进 | 主动发现腐化，及时重构 | 记录 ADR，标注临时方案 |

## Part B：AI 协作工作流

| # | 工作流 | 一句话 | 核心习惯 |
|---|--------|--------|---------|
| W1 | 上下文工程 | 给精准上下文得精准代码 | CLAUDE.md < 150 行，任务上下文在 prompt |
| W2 | 可验证增量 | 小切片，做完验证再推进 | 垂直切片 → 测试 → commit → /clear；无依赖的 Task 可并行 |
