# 代码审查工作流 Skill

AI 生成的代码有系统性偏见——逻辑错误率 1.7x、安全漏洞率 2.74x、happy path 偏见、爱加未要求的功能。同一个会话对自己刚写的代码有认知盲区。本 Skill 通过 Writer/Reviewer 分离和多维度审查来对抗这些偏见。

本 Skill 是 `architecture-principles` 工作流 W3（审查与学习循环）的**具体操作手册**——W3 给出三问快速自检和学习循环理念，本 Skill 告诉你怎么做一次完整的审查。

## 三个阶段

| 阶段 | 做什么 | 典型轮次 |
|------|--------|---------|
| 1. 准备 | 新会话，读 diff，理解变更意图 | 1 轮 |
| 2. 多维度审查 | 按 5 个维度逐项检查 | 2-4 轮 |
| 3. 沉淀 | 记录发现，更新规则 | 1 轮 |

**在 Skill 链条中的位置：**

```
dev-workflow 阶段 5（commit）
       ↓
 code-review（阶段 1-3）  ← 你在这里（新会话）
       ↓
 错误模式 → 写入 CLAUDE.md 禁止项
```

---

## 阶段 1：准备

**什么时候做：** Task commit 之后。必须在**新会话**中执行（Writer/Reviewer 分离）。

**核心原则：** 审查者不应该带着写代码时的思维惯性。新会话 = 新视角。

**执行步骤：**

1. **切换上下文**：开一个新终端，或在当前会话执行 `/clear`
2. **读取变更**：
   ```
   git diff HEAD~1    # 单个 commit
   git diff main      # 整个 feature 分支
   ```
3. **读取任务背景**：读 `docs/tasks.md` 中对应的 Task，理解"应该做什么"
4. **形成审查问题**："这个 diff 是否正确、完整、安全地完成了 Task 要求的事，且没有做多余的事？"

> 不要先看代码再想它该做什么。先理解意图，再看代码是否符合意图。

---

## 阶段 2：多维度审查

**什么时候做：** 准备完成后。

**核心原则：** 每个维度独立检查，不要试图一次看完所有问题。

### 5 个审查维度

按以下顺序逐个维度过一遍 diff：

#### 维度 1：正确性

| 检查项 | 关注什么 |
|--------|---------|
| 逻辑正确 | 条件判断、循环边界、短路求值是否正确 |
| 类型安全 | 类型收窄是否充分，有没有隐式 `any` |
| 数据流 | 数据从输入到输出的每一步转换是否正确 |
| 并发/异步 | Promise 有没有遗漏 await，竞态条件 |

#### 维度 2：边界与错误处理

这是 AI 代码最薄弱的环节。

| 检查项 | 关注什么 |
|--------|---------|
| 空值/缺失 | null、undefined、空数组、空字符串 |
| 极端值 | 超长输入、零值、负值、最大整数 |
| 外部调用失败 | 网络超时、服务不可用、返回格式异常 |
| 数据不存在 | 查询返回空、ID 无效、引用被删除 |
| 权限不足 | 未授权访问、过期 token |

> 对照 `change-guard` W2 的完整性检查清单做系统检查。本维度聚焦代码层面，W2 聚焦系统层面（前后端同步、类型一致等），两者互补。

#### 维度 3：安全

| 检查项 | 关注什么 |
|--------|---------|
| 注入 | SQL 注入、命令注入、XSS（用户输入是否经过清洗） |
| 认证/鉴权 | 接口是否检查了权限，敏感操作是否有二次确认 |
| 数据泄露 | 日志中有没有打印敏感信息，错误响应有没有暴露内部细节 |
| 依赖安全 | 新引入的依赖是否可信，版本是否有已知漏洞 |
| 硬编码秘钥 | API key、密码、token 有没有写死在代码里 |

#### 维度 4：范围纪律

AI 最常犯的问题之一——做多了。

| 检查项 | 关注什么 |
|--------|---------|
| 未要求的功能 | diff 中有没有 Task 没要求的新功能 |
| 未讨论的依赖 | 有没有引入 package.json 中原本没有的新依赖 |
| 不必要的重构 | 有没有改动未被修改的代码（"顺手优化"） |
| 过度设计 | 只需要一个简单函数的地方有没有搞出抽象基类/工厂模式 |

#### 维度 5：可维护性

| 检查项 | 关注什么 |
|--------|---------|
| 尺寸 | 文件 > 200 行？函数 > 30 行？ |
| 命名 | 变量/函数名是否清晰表达意图 |
| 分层 | 有没有跨层调用（api 直接调 provider，service 直接操作存储） |
| 重复 | 有没有复制粘贴的代码应该提取为共享函数 |

### 审查节奏

不要逐行看 diff 试图同时找所有问题。按维度分轮：

```
第 1 轮：正确性 + 边界（最重要，花最多时间）
第 2 轮：安全（快速扫描已知模式）
第 3 轮：范围纪律 + 可维护性（快速确认）
```

小改动（< 50 行 diff）可以一轮看完。大改动按上面的节奏分轮。

## 阶段 3：沉淀

**什么时候做：** 审查完成后。

**核心原则：** 审查的价值不只是找到这次的问题，还要防止下次再犯。

### 输出审查报告

```markdown
## 审查报告：[Task X.X / PR #N]

### 发现
| # | 维度 | 严重度 | 文件:行号 | 问题描述 | 建议修复 |
|---|------|--------|-----------|---------|---------|
| 1 | [维度] | 🔴/🟡/ℹ️ | [位置] | [问题] | [修复] |

### 总结
- 🔴 阻塞项（必须修复才能合并）：N 个
- 🟡 建议项（建议修复但不阻塞）：N 个
- ℹ️ 备注（供参考）：N 个

### 模式沉淀（如果发现了重复出现的错误模式）
- [错误模式描述] → 建议加入 CLAUDE.md：[具体规则]
```

### 严重度判定

| 严重度 | 标准 | 举例 |
|--------|------|------|
| 🔴 阻塞 | 会导致运行时错误、数据丢失、安全漏洞 | 未处理的 null 解引用、SQL 注入、硬编码密钥 |
| 🟡 建议 | 不会立即出错但有隐患或违反规范 | 缺少错误处理、函数过长、命名不清 |
| ℹ️ 备注 | 风格偏好或可选优化 | 可以用更简洁的写法、变量名可以更好 |

### 学习循环

审查中发现的**重复出现的错误模式**，抽象为规则写入 CLAUDE.md 的禁止清单：

```
观察：AI 连续两次在 provider 层忘记设置超时
       ↓
抽象：所有外部 HTTP 调用必须设置超时
       ↓
写入 CLAUDE.md：❌ 外部调用不设超时（每个外部调用必须有错误处理和超时）
       ↓
下次 AI 不再犯这个错误
```

> 只沉淀**重复出现的、有代表性的**错误模式。偶发的一次性错误不值得加规则——规则太多反而稀释重要规则的效力。

## 审查粒度选择

不是每次 commit 都需要完整的 5 维度审查。按变更大小选择粒度：

| 变更规模 | 审查方式 | 耗时 |
|---------|---------|------|
| 小改动（< 50 行 diff） | 三问速查：边界？失败？多做了？ | 2 分钟 |
| 中等改动（50-200 行） | 重点审查维度 1-2（正确性 + 边界），扫一眼维度 3-5 | 10 分钟 |
| 大改动（> 200 行 / 涉及多层） | 完整 5 维度分轮审查 | 20+ 分钟 |
| 关键路径改动（认证、支付、数据迁移） | 完整审查 + 安全维度加深 | 30+ 分钟 |

## 反模式

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| 自己审查自己 | 写完代码在同一个会话里 review | 新会话或 /clear 后再审查（Writer/Reviewer 分离） |
| 只看逻辑不看边界 | "逻辑看起来对"就通过 | 边界和错误处理是 AI 代码最薄弱的环节，必须重点看 |
| 审查一切 | 每个 commit 都做完整 5 维度审查 | 按变更规模选择粒度，小改动用三问速查 |
| 只找问题不沉淀 | 找到 bug 修完就忘 | 重复出现的模式写入 CLAUDE.md，防止再犯 |
| 规则膨胀 | 每个发现都加规则 | 只沉淀重复出现的、有代表性的错误模式 |
| 审查变重构 | review 时顺手改了一堆不相关的代码 | 审查只产出报告，修复是另一个 Task |

## 与其他 Skill 的协作

- **dev-workflow**：阶段 5（commit）之后触发 code-review。审查发现的阻塞项回到 dev-workflow 修复。
- **architecture-principles**：W3（审查与学习循环）是本 Skill 的上位框架。W3 的三问速查是本 Skill 的轻量模式。
- **change-guard**：W2（完整性检查）聚焦系统层面（前后端同步、类型一致），本 Skill 聚焦代码层面（逻辑、边界、安全）。两者互补，不重复。W3（Bug 排查）在审查发现问题后可用于深度调查。
- **CLAUDE.md**：阶段 3 的学习循环产出写入 CLAUDE.md 禁止清单，形成持续改进闭环。
