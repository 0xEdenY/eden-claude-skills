# Skill 管理工具

管理 skill 的四个工作流：日常查看、质量维护、新增审查、全局理解。

## 四个工作流

| 工作流 | 什么时候用 | 做什么 |
|--------|-----------|--------|
| 列表 | "有哪些 skill"、"查找 skill" | 列出所有已安装 skill |
| 健康检查 | "检查 skill"、"skill 有没有问题" | 逐项检查格式、引用、规范 |
| 兼容性审查 | "这个 skill 兼容吗"、安装新 skill 后 | 检查新 skill 和现有体系的冲突 |
| 链条可视化 | "skill 全景"、"skill 怎么串联" | 输出 skill 依赖关系和链条图 |

---

## 工作流 1：列表

**触发词：** "有哪些 skill"、"查找 skill"、"能做什么"

**执行步骤：**

1. 列出全局 skill 目录：`ls ~/.claude/skills/`
2. 列出项目级 skill 目录（若存在）：`ls .claude/skills/ 2>/dev/null`
3. 读取每个 skill 的 `SKILL.md` frontmatter，提取 `name` 和 `description`
4. 以表格形式展示：名称 + 一句话描述 + 作用域（全局/项目）
5. 如果用户有关键词，按描述相关性排序推荐

**输出格式：**

```markdown
## 已安装的 Skills

| Skill | 描述 | 作用域 |
|-------|------|--------|
| [name] | [description 的前 30 字] | 全局/项目 |

## 使用方法
输入 `/skill-name` 或直接描述你的需求，Claude 会自动匹配合适的 skill。
```

## 工作流 2：健康检查

**触发词：** "检查 skill"、"skill 健康"、"skill 有没有问题"

**执行步骤：**

1. 遍历所有已安装 skill 的 `SKILL.md`
2. 逐项执行以下检查
3. 输出分级报告

**第一步：判定 skill 类型**

| 类型 | 特征 | 举例 |
|------|------|------|
| 链条型 | 有上下游依赖，在工作流中和其他 skill 串联 | dev-workflow、change-guard、product-builder |
| 工具型 | 独立使用，不依赖也不被依赖于工作流链条 | humanizer、skill-admin |

类型不同，检查标准不同。不要用链条型的标准去要求工具型 skill。

**第二步：通用检查（所有 skill）**

| 检查维度 | 具体检查 | 严重度 |
|---------|---------|--------|
| Frontmatter 完整性 | `name` 和 `description` 存在且非空 | ❌ 错误 |
| 触发词 | description 中包含触发场景描述 | ⚠️ 警告 |
| 交叉引用完整性 | 引用的 skill 名在已安装列表中存在 | ❌ 错误 |
| 交叉引用准确性 | 引用的工作流编号/原则编号在目标 skill 中存在 | ⚠️ 警告 |
| 尺寸 | 行数在 30-500 行之间 | ⚠️ 警告 |
| 过拟合 | 不包含项目特定的文件路径、类名、函数名 | ⚠️ 警告 |
| 过度指导 | 是否存在"模型已能自行判断但仍在显式要求"的指令 | ⚠️ 警告 |

**第三步：链条型 skill 额外检查**

| 检查维度 | 具体检查 | 严重度 |
|---------|---------|--------|
| 协作章节 | 有"与其他 Skill 的协作"章节，且内容非空 | ⚠️ 警告 |
| 快速参考 | 有"快速参考"或总结性章节 | ⚠️ 警告 |
| 链条位置 | 明确说明了前置和后续 skill | ⚠️ 警告 |

> 工具型 skill 不要求协作章节和链条位置——有则更好，没有不扣分。语言不要求统一——外部安装的英文 skill 在中文体系中是正常的。

**交叉引用检查方法：**

```
1. 在每个 skill 中扫描 `参考 [skill-name]`、`见 [skill-name]`、
   `[skill-name] W[N]`、`[skill-name] 原则 [N]`、
   `[skill-name] 工作流 [N]`、`[skill-name] 阶段 [N]` 等模式
2. 检查 [skill-name] 是否在已安装列表中
3. 如果引用了编号（W1、原则 4、阶段 3），读取目标 skill 确认该编号存在
```

**过拟合检查方法：**

```
扫描以下模式（出现则标记为疑似过拟合）：
- 具体文件路径：匹配 `src/` 或 `packages/` 开头的路径
- 具体类名/函数名：匹配 camelCase 或 PascalCase 且在代码块外的标识符
- 排除误判：
  - 文档约定路径（docs/tasks.md、docs/prd.md）不算过拟合
  - 代码块内的示例不算（除非示例本身过于具体）
  - skill 链约定的引用不算（change-guard W1 等）
```

**输出格式：**

```markdown
## Skill 健康检查报告

### ✅ 通过（N 个）
[skill-name], [skill-name], ...

### ⚠️ 需关注（N 个）
- **[skill-name]**
  - [警告] 第 X 行：[具体问题]
  - [建议] [修复建议]

### ❌ 问题（N 个）
- **[skill-name]**
  - [错误] [具体问题]
  - [修复] [具体操作]
```

## 工作流 3：兼容性审查

**触发词：** "新 skill 兼容吗"、"审查 [skill-name]"、"这个 skill 和现有的冲突吗"

**什么时候用：** 安装外部 skill 后，或创建新 skill 后。

**输入：** 指定一个 skill 名作为被审查对象。

**执行步骤：**

1. 读取被审查 skill 的完整内容
2. 读取所有现有 skill 的 description 和关键章节
3. 逐维度对比
4. 输出审查报告

**审查维度：**

| 维度 | 做什么 | 怎么判定 |
|------|--------|---------|
| 触发词冲突 | 提取双方 description 中的触发词，检查重叠 | 同一个词触发两个 skill → 标记，分析是否真的冲突还是互补 |
| 职责重叠 | 对比双方覆盖的工作流/阶段 | 两个 skill 都包含"影响分析"类内容 → 标记 |
| 指令冲突 | 检查是否有矛盾的规则 | A 说"测试一切" + B 说"按需测试" → 标记 |
| 链条位置 | 判断新 skill 应在链条中的哪个位置 | 基于输入/输出和依赖关系推断 |
| 格式一致性 | 检查 frontmatter、语言、章节结构是否和现有 skill 一致 | 不一致 → 标记 |
| 与 CLAUDE.md 一致性 | 检查新 skill 的规则是否和项目 CLAUDE.md 矛盾 | 如果有 CLAUDE.md，读取并对比关键规则 |

**触发词冲突的处理原则：**

触发词重叠不一定是问题。关键看职责是否互补：
- **互补**：change-guard 和 dev-workflow 都被"重构"触发，但一个管影响分析，一个管执行 → 不冲突
- **冲突**：两个 skill 都包含"怎么写测试"的完整指导 → 需要合并或划清边界

**输出格式：**

```markdown
## 兼容性审查：[skill-name]

### 触发词分析
- "[触发词]" → 同时触发 [skill-A] 和 [skill-B]
  → [互补/冲突]：[原因]

### 职责边界
- ✅ 和 [skill-name] 无重叠
- ⚠️ 和 [skill-name] 在 [领域] 有交叉 → [建议]

### 指令一致性
- ✅ [规则] 和 CLAUDE.md 一致
- ❌ [规则] 和 [skill-name] 的 [规则] 矛盾 → [建议]

### 链条位置
[前置 skill] → [被审查 skill] → [后续 skill]

### 格式一致性
- ✅ / ⚠️ [具体问题]
```

## 工作流 4：链条可视化

**触发词：** "skill 全景"、"skill 怎么串联"、"画 skill 关系图"

**执行步骤：**

1. 读取所有 skill 的"与其他 Skill 的协作"章节
2. 解析引用关系（A 引用了 B 的哪个部分）
3. 构建依赖图
4. 检测问题（孤立 skill、断裂引用、循环依赖）
5. 输出可视化图 + 问题报告

**引用关系解析：**

从每个 skill 中提取：
- **输入依赖**：本 skill 需要读取哪些 skill 的产出
- **输出被依赖**：本 skill 的产出被哪些 skill 引用
- **规则依赖**：本 skill 引用了哪些 skill 的原则/工作流作为理论依据

**输出格式：**

```markdown
## Skill 链条全景

### 依赖关系图

[文本格式的依赖图，按生命周期分区]

### 引用矩阵

| 被引用 ↓ / 引用者 → | skill-A | skill-B | skill-C |
|---------------------|---------|---------|---------|
| skill-A             | —       | W1, W2  | 原则 1  |
| skill-B             |         | —       |         |
| skill-C             |         |         | —       |

### 问题检测
- 🔴 [skill-name] 引用了不存在的 [target-skill]
- 🟡 链条型 skill 缺少上游或下游连接
- 🟡 [链条位置] 存在断裂（从 X 到 Y 之间没有 skill 覆盖）
- ℹ️ 工具型 skill 孤立是正常的，不报警
```

## Skill 生命周期管理

### 创建准入标准

创建新 Skill 之前必须满足全部 4 条：

1. **有实证失败** — 能指出具体的、反复出现的失败模式（不是假设的风险）
2. **CLAUDE.md 不够** — 如果一条规则就能解决，写 CLAUDE.md 而不是建 Skill
3. **不与现有 Skill 重叠** — 触发条件不能跟已有 Skill 冲突或模糊
4. **有验收方式** — 能说清"有了这个 Skill 后，之前的失败应该不再发生"

不满足的，先作为 CLAUDE.md 规则试行。反复证明有效后再升级为 Skill。

### 退役标准

退役信号：模型升级后、长期未触发、频繁修补边界情况、与其他 Skill 职责模糊。

退役流程：重命名 `SKILL.md` 为 `SKILL.md.disabled` → 工作 1-2 周观察 → 没问题则删除 → 有问题则恢复并记录存在理由。

## Skill 设计原则

创建或引入新 skill 时的指导原则。

### 原则 1：Skill 管方法论，CLAUDE.md 管约束

Skill 描述**通用的工作流和判断标准**（什么时候测试、怎么分层实现）。项目特定的约束（用什么技术栈、什么命名规范、哪些逻辑必须测试）由 CLAUDE.md 提供。两者在运行时合并为具体指导。

```
Skill（通用）：  "核心逻辑必须有测试"
CLAUDE.md（具体）："核心逻辑 = 实体解析、AI 输出校验、去重、信号匹配、通知路由"
合并效果：      AI 知道这 5 类逻辑必须写测试
```

**推论：** Skill 中不应出现项目特定的文件路径、类名、函数名、技术栈名称。代码示例用 `[占位符]` 或伪代码。

### 原则 2：先定位，再设计

新 skill 的第一个问题不是"写什么内容"，而是"在链条中站哪个位置"。明确：
- **输入**：本 skill 需要哪些 skill 的产出？
- **输出**：本 skill 的产出被谁消费？
- **不做什么**：哪些职责已被现有 skill 覆盖？

避免和现有 skill 职责重叠。如果新 skill 的某个章节和已有 skill 内容相似，引用它而不是复制它。

### 原则 3：区分链条型和工具型

| | 链条型 | 工具型 |
|--|--------|--------|
| 特征 | 有上下游依赖 | 独立使用 |
| 必须有 | 协作章节、链条位置、快速参考 | Frontmatter、触发词 |
| 语言 | 和体系一致 | 不强制 |
| 举例 | dev-workflow, change-guard | humanizer, skill-admin |

设计时先判定类型，再决定需要哪些章节。不要用链条型的标准要求工具型 skill。

### 原则 4：从外部 skill 吸收而非照搬

引入社区 skill 时，做选择性改造：
1. **吸收**核心理念（如 TDD 的红绿循环）
2. **丢弃**和现有体系冲突的部分（如硬性覆盖率要求）
3. **扩展**社区 skill 不具备但你需要的能力（如任务类型分流）

直接安装未经审查的外部 skill 会引入指令冲突。

### 原则 5：格式跟随体系

和现有 skill 保持格式一致：
- Frontmatter：`name`（kebab-case）+ `description`（含触发词）
- 语言：中文主体 + 英文技术术语（链条型 skill）
- 章节：开篇哲学 → 工作流/阶段 → 反模式 → 协作 → 快速参考
- 模板用 `[占位符]`，不用具体值

## 反模式

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| 检查强迫症 | 每次对话都跑一遍健康检查 | 新增/修改 skill 后才跑，日常不需要 |
| 形式大于内容 | 为了通过格式检查强行补章节，但内容空洞 | 章节要有实质内容，宁可标注"暂不适用"也不凑字数 |
| 过度拆分 | 为了避免触发词冲突把一个 skill 拆成三个 | 触发词重叠但职责互补是正常的，不需要拆 |
| 忽略 CLAUDE.md | 只检查 skill 之间的一致性，不检查和 CLAUDE.md 的一致性 | CLAUDE.md 是最高优先级，skill 不能和它矛盾 |

## 与其他 Skill 的协作

- **所有 skill**：工作流 2（健康检查）和工作流 3（兼容性审查）需要读取所有已安装 skill 的内容。
- **architecture-principles**：链条可视化中的依赖关系分析基于各 skill 的交叉引用，architecture-principles 通常是依赖图的根节点。
- **CLAUDE.md**：兼容性审查的"指令一致性"维度需要读取项目的 CLAUDE.md 文件。
- **excalidraw-diagram**：工作流 4（链条可视化）输出文本格式关系图；如果需要更精美的交互式图，可以用 excalidraw-diagram 生成。
