# UX 设计原则与检查清单

面向有设计基础的开发者。不教基础概念，提供可直接使用的检查框架。

## 使用方式

三种场景：
- **设计时**：做界面决策时参考原则
- **审查时**：UI 完成后用检查清单逐项过
- **修复时**：用户反馈"不好用"时定位问题

---

## Part A：7 条设计原则

基于 Nielsen 启发式评估重新组织，合并了重叠项，补充了 AI 生成 UI 的特有问题。

### 原则 1：状态透明（System Status Visibility）

**核心：** 用户在任何时刻都应该知道——系统在干什么、我在哪里、我的操作有没有生效。

**检查项：**
- [ ] 加载中有反馈吗？（spinner、skeleton、进度条）
- [ ] 操作成功有确认吗？（toast、状态变化、动画）
- [ ] 操作失败有明确提示吗？（不是空白，不是控制台报错）
- [ ] 长操作有进度指示吗？（上传、导出、批量操作）
- [ ] 当前位置清晰吗？（面包屑、高亮的导航项、页面标题）
- [ ] 表单提交后用户知道发生了什么吗？

**AI 常犯的错：** 生成的页面缺少加载状态和错误状态——只做了 happy path 的 UI。让 AI 写界面时，明确要求同时处理四种状态：正常、加载中、空数据、错误。

---

### 原则 2：说人话（Match Real World）

**核心：** 用用户的语言，不用技术术语。信息的组织方式要符合用户心智模型，不是系统的数据结构。

**检查项：**
- [ ] 标签和按钮用的是用户能理解的词吗？（"提交订单" vs "POST request"）
- [ ] 信息分组符合用户的思维方式吗？（而不是数据库表的结构）
- [ ] 日期、金额、单位的格式符合目标用户的习惯吗？
- [ ] 错误提示是用户能理解的吗？（"网络连接失败，请稍后重试" vs "Error 500"）
- [ ] 图标和隐喻是通用可理解的吗？

**AI 常犯的错：** 用技术术语命名 UI 元素（"Entity List"而不是"客户列表"），错误信息暴露技术细节（"TypeError: Cannot read property 'name' of undefined"）。

---

### 原则 3：用户掌控（User Control & Freedom）

**核心：** 用户会犯错。提供明确的撤销、返回、取消方式。不要把用户锁在无法退出的状态里。

**检查项：**
- [ ] 每个操作都可以撤销或取消吗？（特别是破坏性操作）
- [ ] 破坏性操作有二次确认吗？（删除、覆盖、不可逆变更）
- [ ] 多步骤流程可以返回上一步吗？
- [ ] 模态框/弹窗可以关闭吗？（ESC 键、点击背景、关闭按钮）
- [ ] 表单填到一半离开会丢失数据吗？（应该提醒或自动保存）
- [ ] 有明确的"退出"路径吗？（不是只有前进没有后退）

**AI 常犯的错：** 生成的表单没有取消按钮，弹窗只有确认没有取消，多步骤流程不能回退。

---

### 原则 4：一致性（Consistency & Standards）

**核心：** 同样的元素在不同地方的外观和行为应该一致。遵循平台约定，不发明新的交互模式。

**检查项：**
- [ ] 同一个操作在不同页面用同样的措辞吗？（不要一会儿"保存"一会儿"提交"一会儿"确认"）
- [ ] 按钮风格一致吗？（主要操作、次要操作、危险操作有统一的视觉区分）
- [ ] 间距、字号、颜色有系统吗？（不是每个页面各自为政）
- [ ] 链接看起来像链接吗？按钮看起来像按钮吗？
- [ ] 遵循了平台惯例吗？（移动端的返回手势、Web 的标签页行为）
- [ ] 图标的含义在不同地方一致吗？

**AI 常犯的错：** 每次生成 UI 时从零开始设计样式，不参考已有页面，导致同一个产品里风格不统一。在 CLAUDE.md 中指定设计系统/组件库可以大幅减少这个问题。

---

### 原则 5：预防错误（Error Prevention）

**核心：** 与其写好的错误提示，不如从设计上让错误不容易发生。

**检查项：**
- [ ] 输入有合理的约束吗？（日期选择器而不是手动输入日期、下拉选择而不是自由文本）
- [ ] 容易混淆的操作有视觉区分吗？（"保存草稿"和"发布"用不同颜色/位置）
- [ ] 不可逆操作需要额外确认吗？
- [ ] 表单在提交前做了校验吗？（实时校验 > 提交后报错）
- [ ] 默认值合理吗？（减少用户需要做的选择）
- [ ] 危险按钮和常用按钮有距离吗？（"删除"不要紧挨着"保存"）

**AI 常犯的错：** 所有输入都用 text input，不用 select/datepicker/checkbox 等更受约束的输入方式。不设置合理的默认值。

---

### 原则 6：减少记忆负担（Recognition Over Recall）

**核心：** 让用户认出该做什么，而不是要求他们记住。信息在需要时可见，不要藏起来。

**检查项：**
- [ ] 用户需要记住其他页面的信息才能完成当前操作吗？（如果是，把那个信息带过来）
- [ ] 常用操作是可见的还是藏在菜单深处？
- [ ] 搜索和筛选功能容易找到吗？
- [ ] 最近使用/历史记录有保留吗？
- [ ] 表格或列表中的操作按钮就在该行旁边吗？（而不是需要先选中再去工具栏操作）
- [ ] 复杂表单有内联帮助或占位符提示吗？

---

### 原则 7：克制与聚焦（Minimalist Design）

**核心：** 每个屏幕有一个主要任务，一切为它服务。去掉任何不帮助用户完成当前任务的元素。

**检查项：**
- [ ] 这个页面的核心任务是什么？（只能有一个）
- [ ] 最重要的操作是否最显眼？（视觉层级清晰）
- [ ] 有没有和核心任务无关的元素可以移除或收起？
- [ ] 信息密度合适吗？（不至于压迫，也不至于空洞）
- [ ] 移动端做了适当简化吗？（不是把桌面版硬塞进小屏幕）

**AI 常犯的错：** 把所有信息平铺在一个页面上，没有主次之分。生成的页面往往"什么都有"但"什么都不突出"。

## Part B：审查检查清单

在 UI 完成后做一次系统性审查。可以让 AI 对照这个清单检查，也可以自己过一遍。

### 快速审查（5 分钟，每次 UI 变更后）

```
对照以下清单，检查刚完成的页面/组件：

1. 四种状态都处理了吗？
   □ 正常状态
   □ 加载状态
   □ 空数据状态
   □ 错误状态

2. 用户知道自己在哪、该做什么吗？
   □ 页面标题/面包屑清晰
   □ 主操作按钮明显
   □ 不需要说明书就能用

3. 出错了能恢复吗？
   □ 操作可撤销或可取消
   □ 破坏性操作有确认
   □ 错误提示说人话且提供解决建议
```

### 完整审查（30 分钟，每个迭代/milestone 后）

```
## 导航与结构
- [ ] 信息架构清晰：用户 3 次点击内能到达任何核心功能
- [ ] 导航一致：全局导航在每个页面的行为一致
- [ ] 返回路径清晰：每个深层页面都能回到上一级
- [ ] URL 结构合理：可以从 URL 理解自己在哪

## 表单与输入
- [ ] 标签清晰：每个输入字段有明确的标签
- [ ] 输入约束合理：用合适的输入控件（日期用 datepicker，选择用 select）
- [ ] 实时校验：用户不需要提交后才知道填错了
- [ ] 错误定位：错误提示出现在出错的字段旁边，不只是页面顶部
- [ ] 必填/选填明确标注
- [ ] Tab 键顺序正确

## 反馈与状态
- [ ] 所有异步操作都有加载态
- [ ] 操作成功有明确反馈
- [ ] 错误提示有具体原因和解决建议
- [ ] 空状态有引导（不是空白页面）
- [ ] 长列表有分页或无限滚动

## 视觉与一致性
- [ ] 颜色使用有系统（不超过 3 个主色）
- [ ] 字号层级清晰（标题 > 正文 > 辅助文字）
- [ ] 间距统一（使用 4px/8px 倍数系统）
- [ ] 按钮有明确的视觉层级（主要/次要/危险）
- [ ] 可交互元素有 hover/active 状态

## 响应式与适配
- [ ] 移动端可用（不是桌面版的缩小版）
- [ ] 触摸目标足够大（最小 44px × 44px）
- [ ] 关键流程在移动端能完成
- [ ] 横竖屏切换不会丢失状态

## 可访问性基础
- [ ] 颜色对比度足够（文字至少 4.5:1）
- [ ] 不单独靠颜色传达信息（色盲友好）
- [ ] 图片有 alt 文本
- [ ] 可以只用键盘操作核心流程
- [ ] 焦点状态可见
```

## Part C：AI 生成 UI 的特有问题

AI 生成 UI 时有一些系统性偏差，需要特别检查。

### AI 的 UI 偏差模式

| 偏差 | 表现 | 检查方法 |
|------|------|---------|
| Happy path 偏执 | 只有正常状态的 UI，没有加载、空数据、错误状态 | 检查每个页面的四种状态 |
| 平铺信息 | 所有信息同等权重地铺在页面上 | 检查是否有清晰的视觉层级 |
| 风格漂移 | 每次生成的 UI 样式不一致 | 在 CLAUDE.md 中指定组件库和设计规范 |
| 过度设计 | 加了很多花哨但没用的装饰效果 | 问自己：去掉这个元素用户会受影响吗？ |
| 忽视移动端 | 生成桌面版完美但移动端不可用 | 用浏览器的响应式模式检查 |
| 技术术语泄漏 | 错误信息、标签使用开发者语言 | 让非技术朋友看一眼是否能理解 |

### 在 CLAUDE.md 中预防这些偏差

```markdown
## UI 规范
- 每个页面/组件必须同时处理：正常态、加载态、空数据态、错误态
- 使用 [指定组件库名称] 的组件，不要自行创建新的 UI 样式
- 颜色方案：[列出主色、辅助色、语义色]
- 间距系统：使用 4px 的倍数
- 错误信息面向用户编写，不暴露技术细节
- 所有页面必须在 375px 宽度下可用
```

## Part D：视觉设计品质（反 AI 审美）

基于 Anthropic 官方 frontend-design skill 的核心理念，过滤为适用于专业工具型产品的版本。

### 问题

AI 生成的 UI 有一种可识别的"AI 审美"：Inter 字体、紫色渐变白底、均匀分布的卡片网格、无性格的配色。这些不是错误——而是缺乏意图。

### 核心原则：有意图地选择，而非接受默认

不需要"大胆"或"极端"——但每一个视觉选择都应该是**有理由的**，而不是"AI 默认生成的"。

### 5 个视觉维度

#### 1. 字体意图

| 场景 | 做法 | 避免 |
|------|------|------|
| 数据密集型界面 | 选择高可读性的等宽/半等宽字体用于数字和代码 | 全局使用 Inter/Roboto |
| 标题/导航 | 用一个有辨识度的无衬线字体建立品牌感 | 和正文用完全相同的字体 |
| 正文 | 清晰、中性、阅读舒适 | 装饰性字体用于大段文字 |

关键不是"选罕见字体"，而是**为不同场景有意识地选择不同字体角色**。

#### 2. 色彩体系

```
主色       → 品牌识别、主操作按钮
语义色     → 成功（绿）、警告（黄）、错误（红）、信息（蓝）
数据色     → 图表系列色、分类标识色（需要 5-8 个可区分的颜色）
中性色     → 背景、边框、次要文字（灰色梯度 4-6 阶）
强调色     → 引导注意力的亮色（谨慎使用，一个就够）
```

用 CSS 变量管理。避免"渐变色用于装饰"——渐变用于表达数据含义（热力图、进度）时才有价值。

#### 3. 动效克制

| 值得做的动效 | 不值得做的动效 |
|-------------|---------------|
| 页面/卡片入场的错位渐入（`animation-delay`） | 每个元素都加 `transition` |
| 告警/异常出现时的注意力引导 | 持续跳动的徽章 |
| 状态切换的平滑过渡（展开/折叠） | 纯装饰性的漂浮动画 |
| hover 揭示更多操作 | hover 触发颜色剧变 |

原则：**一个编排好的入场动画 > 散落的微交互**。

#### 4. 空间与布局

- 不是所有内容都要放进卡片——卡片是分组工具，不是默认容器
- 信息密度要**有意识地控制**：仪表盘可以密，详情页应该松
- 留白不是浪费——它引导视线、建立层级
- 表格、列表、卡片混用比全部统一为一种形式更有效

#### 5. 质感与细节

小细节建立品质感：

- 阴影用于表达层级，不是装饰（浅阴影 = 浮起，深阴影 = 弹出）
- 边框的粗细、颜色、圆角要有系统（不是每个组件各自随机）
- 图标风格统一（线条图标或填充图标，不要混用）
- 空状态有设计（插图或提示文案），不是空白

### 视觉品质快速自检

```
完成一个页面后，问自己 3 个问题：

1. 能认出这是你的产品吗？
   → 如果换掉 logo 后看不出区别，说明缺乏视觉身份

2. 有哪个视觉选择是"AI 默认给的"？
   → 逐一审视字体、颜色、间距、阴影——每个都应该能说出选择的理由

3. 去掉一个视觉元素后页面会变差吗？
   → 如果去掉后无感，说明那个元素是装饰噪音，应该去掉
```

## 与其他 Skill 的协作

- **product-builder**：阶段 2（PRD）的页面/界面定义步骤产出的界面描述，后续开发时用本 Skill 的检查清单验证
- **architecture-principles**：原则 2（显式约束）中应包含 UI 规范（组件库、颜色、间距），原则 4（数据模型优先）提醒不要用数据库结构决定 UI 信息架构
- **change-guard**：工作流 2（完整性检查）的前端部分，可以引用本 Skill 的快速审查清单
- **product-iteration**：阶段 1（收集信号）的定性反馈中，用户说"不好用"时用本 Skill 的原则定位具体问题
