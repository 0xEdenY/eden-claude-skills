# 开发执行工作流 Skill

AI 编程最常见的失败不是写错代码，而是按错顺序写代码——先写 UI 再补后端，先写实现再想测试，先动手再理解需求。本 Skill 给出从接到任务到 commit 的完整执行节奏，确保每一步都在正确的位置。

本 Skill 是 `architecture-principles` 工作流 W2（可验证的增量交付）的**具体操作手册**——W2 给出 8 步抽象节奏，本 Skill 告诉你每一步具体怎么做。

## 五个阶段

| 阶段 | 做什么 | 典型轮次 |
|------|--------|---------|
| 1. 理解任务 | 读任务、读规格、确认范围、判定任务类型 | 1-2 轮 |
| 2. 规格确认 | 补全验收标准和边界情况 | 1-2 轮 |
| 3. 测试先行 | 为核心逻辑写测试（红灯状态） | 2-3 轮 |
| 4. 分层实现 | 按任务类型对应的顺序逐层实现，让测试变绿 | 3-6 轮 |
| 5. 验收提交 | 全量测试 → 完整性检查 → commit | 1 轮 |

**在 Skill 链条中的位置：**

```
change-guard W1（影响分析）
       ↓
 dev-workflow（阶段 1-5）  ← 你在这里
       ↓
change-guard W2（完整性检查，在阶段 5 中调用）
```

## 阶段 1：理解任务

**什么时候做：** 每个 Task 的第一步。在 change-guard W1（影响分析）之后。

**核心原则：** 动手之前完全理解"做什么"和"做到什么程度"。

**执行步骤：**

1. **读任务定义**（`docs/tasks.md` 中对应的 Task）
2. **读 PRD 相关章节**（`docs/prd.md` 中任务引用的需求）
3. **读相关代码**（如果是修改现有功能，先读要改的文件）
4. **判定任务类型**——不同类型决定阶段 3-4 的执行路径：

| 任务类型 | 特征 | 阶段 3 测试策略 | 阶段 4 实现顺序 |
|---------|------|----------------|----------------|
| 新功能（后端） | 新 API、新 service、新数据模型 | 为核心逻辑写测试 | schema → 查询 → service → API route |
| 新功能（全栈） | 后端 + 前端都要改 | 为核心逻辑写测试 | 后端全部 → 共享类型 → 前端 |
| 新功能（纯前端） | 只改前端，不改后端 | 按需（有交互逻辑才测） | 共享类型 → 组件 → 页面 → hooks |
| 重构 | 改结构不改行为 | **先补测试确认基线** | 小步改，每步跑测试 |
| Bug 修复 | 改行为 | **先写失败测试复现 bug** | 改代码让测试变绿 |

> 重构和 Bug 修复**强制测试先行**——没有测试就没有安全网。新功能只需测试核心逻辑。

## 阶段 2：规格确认

**什么时候做：** 任务的验收标准不够清晰或缺少边界情况时。如果 Task 已有完整的 Given-When-Then 验收标准且覆盖了边界情况，可以跳过。

**核心原则：** 不补全验收标准就不动手写代码。

**需要追问的场景（对话式，1-2 轮）：**
- 边界情况："这个功能的边界有哪些？空输入、超长输入、并发？"
- 失败处理："如果外部依赖挂了怎么办？报错还是降级？"
- 数据兼容："这个改动影响现有数据吗？老数据需要迁移吗？"
- 范围确认："这个 Task 只做 X，不做 Y，对吗？"

**规格确认模板：**

```markdown
## Task X.X 规格确认

### 核心行为
- Given [前置条件], When [用户操作], Then [期望结果]
- Given [前置条件], When [用户操作], Then [期望结果]

### 边界情况
- Given [边界条件], When [操作], Then [处理方式]

### 错误处理
- Given [失败条件], When [操作], Then [降级/报错方式]

### 不做的事（明确排除）
- [不在本 Task 范围内的内容]
```

## 阶段 3：测试先行

**什么时候做：** 规格确认后、写实现代码之前。

**核心原则：** 不是"测试一切"，而是"测试值得测试的"。测试的目的是用机器验证规格，不是追求覆盖率数字。

### 什么值得测试

| 代码特征 | 为什么测 | 举例 |
|---------|---------|------|
| 纯函数 + 有分支逻辑 | 投入产出比最高，不需要 mock | 匹配算法、去重逻辑、格式转换 |
| 数据转换 / 解析 | 边界情况多，人肉验证容易漏 | schema 校验、外部 API 响应解析 |
| 条件匹配 / 路由 | 组合爆炸，需要系统验证 | 规则引擎、权限判定、通知分发 |
| 状态机 / 生命周期 | 状态转换规则容易出错 | 订单状态流转、审批流程 |

### 什么不需要测试

| 代码类型 | 为什么不测 |
|---------|----------|
| 薄包装层（API route 只做参数校验和转发） | Zod schema 本身就是验证，route 逻辑太薄没有测试价值 |
| 简单 CRUD（无业务逻辑的增删改查） | 测的是 ORM，不是你的代码 |
| 纯 UI 布局（无交互逻辑） | 用眼睛看比写测试效率高 |
| 第三方库封装层（只做调用转发） | 测的是别人的代码 |

### 测试编写规范

- 文件位置：与被测文件同目录（`foo.ts` → `foo.test.ts`）
- 命名规则：`should [行为] when [条件]`——描述行为，不描述实现
- 结构：Given（准备数据）→ When（执行操作）→ Then（验证结果）

```typescript
// [被测文件名].test.ts，与被测文件同目录

describe('[模块名]', () => {
  it('should [期望行为] when [条件]', () => {
    // Given
    const input = /* 准备测试数据 */;

    // When
    const result = doSomething(input);

    // Then
    expect(result).toEqual(/* 期望输出 */);
  });

  it('should [另一个行为] when [另一个条件]', () => {
    // ...
  });
});
```

### Mock 策略

只 mock **层边界**，不 mock 内部实现：

| 被测层 | Mock 什么 | 不 Mock 什么 |
|-------|----------|-------------|
| 服务层 | 数据查询函数、外部 API 调用 | 服务内部的纯函数 |
| 外部封装层 | HTTP 请求、第三方 SDK | 内部数据转换 |
| 纯函数 | 什么都不 mock | — |

```typescript
// 正确：mock 层边界（数据层、外部依赖）
mock('数据查询模块', () => ({
  findById: fn().mockResolvedValue({ id: '1', title: 'test' }),
}));

// 错误：mock 被测函数的内部帮助函数
// mock('./helpers', () => ({ formatDate: fn() }));
```

**阶段 3 产出：** 一组编译通过但**预期失败**的测试文件（红灯状态）。如果是重构，测试应该在现有代码下**全部通过**（绿灯基线）。

## 阶段 4：分层实现

**什么时候做：** 测试写好之后。

**核心原则：** 自底向上，一层一层让测试变绿。每完成一层跑一次测试。

### 按任务类型的实现顺序

**新功能（后端 / 全栈）：**

```
Step 1：数据模型（如果需要新表/改表）
  → ORM schema 定义
  → 生成并运行迁移
  → 写数据访问函数

Step 2：服务层
  → 业务逻辑
  → 这一步完成后，阶段 3 的测试应该变绿 ✅

Step 3：入口层
  → API 路由
  → 请求校验 schema
  → 错误处理映射（service error → HTTP status）

Step 4：共享类型（如果前端也要改）
  → 更新共享类型定义

Step 5：前端（如果是全栈任务）
  → Store / hooks → 组件 → 页面
  → 每个组件处理四种状态（参考 ux-principles）：正常、加载中、空数据、错误
```

**重构：**
```
Step 1：确认测试基线（阶段 3 已完成，测试应该全绿）
Step 2：小步修改，每步跑测试
Step 3：测试结果不变 = 重构成功
```

**Bug 修复：**
```
Step 1：复现测试已写好（阶段 3，红灯）
Step 2：定位并修复代码
Step 3：复现测试变绿 + 全量测试不回归
```

### 层过渡检查

每写完一层、进入下一层之前，快速检查：

| 过渡 | 检查项 |
|------|--------|
| db → service | 查询返回的类型和 service 期望的一致吗？ |
| service → API | service 的所有错误类型都有对应的 HTTP status 映射吗？ |
| 后端 → 前端 | 共享类型更新了吗？API 返回格式和前端期望一致吗？ |

### 尺寸看门

写代码过程中持续检查：
- 文件 > 200 行 → 拆分
- 函数 > 30 行 → 提取子函数

---

## 阶段 5：验收提交

**什么时候做：** 代码写完了。

**核心原则：** commit 之前做三件事——跑测试、查完整性、写 commit message。

**执行清单：**

```
1. 全量测试
   → pnpm test
   → 全绿才继续；有红灯先修

2. 完整性检查（交给 change-guard W2）
   → 类型一致？前后端同步？测试覆盖？做了没要求的事？
   → 简单改动用 6 问速查版

3. Commit
   → Conventional Commits：feat: / fix: / refactor: / test:
   → 每个 Task 一个 commit，不攒
   → git add [具体文件]（不要 git add .）

4. 切换上下文
   → /clear（清理 AI 上下文，避免前一个任务的记忆污染）
   → 开始下一个 Task
```

## 反模式

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| 顺序反了 | "我先把页面做出来看看效果" | 数据模型 → 服务层 → 入口层 → 界面，顺序不能反 |
| 测试一切 | 给每个 CRUD 查询、每个 UI 组件写测试 | 只测核心逻辑——有分支、有转换、有条件匹配的代码 |
| 不测试 | "这个太简单了不需要测试" | CLAUDE.md 明确要求的 5 类逻辑必须测 |
| 大步走 | 一口气写完所有层再跑测试 | 每写完一层跑一次测试 |
| 跳过规格 | 看任务标题就开始写代码 | 先读 PRD 对应章节，确认验收标准 |
| Mock 一切 | 把所有依赖都 mock 掉 | 只 mock 层边界，纯函数不 mock |
| 攒 commit | 三个 Task 的代码一起提交 | 每个 Task 完成后立即 commit |
| 测实现细节 | 测试断言内部变量值或调用次数 | 测试行为和输出，不测实现方式 |

## 与其他 Skill 的协作

- **change-guard**：阶段 1 之前先用 W1 做影响分析确认变更范围；阶段 5 中调用 W2 做完整性检查。dev-workflow 是两者之间的执行层。
- **architecture-principles**：原则 1（规格先行）是阶段 2-3 的理论依据；原则 4（数据模型优先）是阶段 4 实现顺序的依据；原则 5（清晰分层）决定了分层实现策略；W2（可验证增量）是本 Skill 的上位框架。
- **product-builder / product-iteration**：阶段 1 读取它们产出的 `docs/tasks.md` 和 `docs/prd.md`。验收标准格式沿用 product-builder 阶段 4 的 Given-When-Then 模板。
- **ux-principles**：阶段 4 前端实现步骤引用其四种状态检查（正常、加载中、空数据、错误）。
- **design-system.md**：阶段 4 涉及前端时，必须先读项目的 `docs/design-system.md`（如果存在），确保配色、字体、间距、组件风格和已有页面一致。
