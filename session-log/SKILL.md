# 会话记录 Skill

保存当前会话的完整对话记录。不是摘要——是能让下一个会话还原所有决策上下文的详细记录。

## 类型

工具型 skill。不属于任何工作流链条，但和所有 skill 配合——任何 skill 驱动的长会话都可能需要在结束前保存记录。

## 触发条件

| 触发方式 | 条件 |
|---------|------|
| 用户手动 | "保存记录"、"写 session log"、"收工"、"下次继续"、"保存进度" |
| Claude 主动 | 对话超过 30 轮且产出了重要决策（见 CLAUDE.md 规则） |
| 系统提示 | 收到上下文压缩警告时，先保存再继续 |

## 执行步骤

### Step 1：确定文件

文件名：`docs/session-log-{YYYY-MM-DD}.md`

- 如果今天的文件已存在，用 `---` 分隔追加新会话段落
- 如果不存在，创建新文件

### Step 2：读取格式参考

读取项目中最近一个 `docs/session-log-*.md` 文件的前 100 行，理解当前项目的 session-log 格式风格。不同项目可能有不同风格——**跟随已有格式，不强制统一模板**。

如果项目中没有 session-log 文件，使用以下默认结构：

```
# 标题（描述会话做了什么）

**日期**：
**参与者**：
**性质**：一句话
**产出**：概要列表

---

## 阶段 N：[主题名]

### N.1 [子主题]

[讨论过程]

---

## 产出文件清单
## 关键设计决策索引
```

### Step 3：回顾对话并写入

从对话开头到当前位置，提取并记录以下内容：

**必须记录（不可省略）：**

| 内容 | 怎么记录 | 例子 |
|------|---------|------|
| 用户原话 | `> Eden: [原话]` 格式引用 | `> Eden: 行动者不一定掌握定价权啊` |
| 苏格拉底对话过程 | 记录每个 Q 和用户的选择 | `Q: 先有范式还是先有叙事？ → Eden: B` |
| Claude 的独立分析 | 当用户要求"独立冷静客观分析"时的结论 | Claude 发现 4 个问题：1... 2... |
| 设计决策 | 背景 + 讨论过程 + 最终选择 + 理由 | 行动者→参与者，因为影响力是光谱 |
| 被否决的方案 | 方案内容 + 否决原因 | 第一版补丁式修改被否，因为过拟合 |
| 具体例子 | 例子的实际内容，不是"举了个例子" | KOL 在叙事早期是驱动力、末期是情绪指标 |
| 产出文件 | 文件路径 + 一句话描述 | docs/parallax/design-system.md — 设计规范 |

**不需要记录：**
- 纯操作性的来回（"帮我读一下这个文件"→ 读完 → "好"）
- 工具调用的技术细节（git 命令、文件编辑的具体行号）
- 重复的确认（"继续" → "好"这种不含新信息的交互）

**记录深度标准：** 一个没参与本次会话的人，读完 session-log 后应该能：
1. 理解每个决策的 why（不只是 what）
2. 知道哪些方案被考虑过、被否决过
3. 还原苏格拉底对话的关键推理链
4. 找到所有产出文件

### Step 4：提炼 memory（如有需要）

写完 session-log 后，检查对话中是否有应该持久化到 memory 的内容：

| memory 类型 | 什么时候提炼 | 例子 |
|------------|------------|------|
| user | 发现了用户新的角色/偏好/知识 | 用户喜欢 OKX 浅色风格 |
| feedback | 用户纠正了 Claude 的做法 | "补丁式修改被拒绝"、"UI 必须生成网页" |
| project | 发现了项目的重要决策/状态 | 概念字典从 Actor 改为 Participant |
| reference | 发现了外部资源引用 | — |

对照 `memory/MEMORY.md` 现有索引，**不重复已有 memory**。只提炼本次会话中**新发现的**、**未来会话需要知道的**内容。

提炼后：写入 memory 文件 + 更新 MEMORY.md 索引。

### Step 5：提交

```bash
git add docs/session-log-{date}.md
git add memory/  # 如果有 memory 更新
git commit -m "docs: session log for {date} — {一句话描述}"
```

## 质量自检

写完后对照检查：

- [ ] 每个重要决策都记录了背景+过程+结论吗？
- [ ] 用户否定/纠正 Claude 的地方都记录了吗？
- [ ] 关键例子的**具体内容**记录了吗？（不是"Claude 举了个例子"）
- [ ] 产出文件清单完整吗？
- [ ] 一个没参与会话的人能读懂每个决策的 why 吗？
- [ ] 需要提炼到 memory 的内容都提炼了吗？
- [ ] 没有和已有 memory 重复吗？

## 反模式

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| 写摘要而非记录 | "讨论了 X，决定了 Y" | 记录讨论过程：问了什么、选了什么、为什么 |
| 省略否决方案 | 只记录最终方案 | 被否决的方案和否决原因同样重要 |
| 例子变成指代 | "Claude 用了一个例子说明" | 记录例子的实际内容 |
| 每轮都写 log | 5 轮对话就写一次 | 30+ 轮或有重要决策时才写 |
| 只写 log 不写 memory | session-log 越来越长但 memory 不更新 | 跨会话有用的决策必须提炼到 memory |

## 与其他 Skill 的协作

- **所有 skill**：任何 skill 驱动的长会话都可能触发 session-log。session-log 记录的内容包括 skill 执行过程中的决策。
- **product-builder / product-iteration**：产品决策会话的 session-log 尤其重要——概念定义、PRD 变更、任务拆分的决策链需要完整记录。
- **skill-admin**：session-log 是工具型 skill，不在链条中。skill-admin 健康检查时按工具型标准检查。

## 快速参考

| 步骤 | 做什么 | 产出 |
|------|--------|------|
| 1. 确定文件 | 检查今天的 session-log 是否存在 | 文件路径 |
| 2. 读取格式 | 读最近的 session-log 参考格式 | 格式模板 |
| 3. 写入记录 | 回顾对话，按深度标准记录 | session-log 文件 |
| 4. 提炼 memory | 检查是否有新的跨会话信息 | memory 文件（如有） |
| 5. 提交 | git add + commit | 提交记录 |
