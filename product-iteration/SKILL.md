# 产品迭代引导 Skill

产品上线后，通过结构化对话引导用户基于真实信号做出迭代决策，避免拍脑袋加功能。

## 与 product-builder 的关系

- **product-builder**：从 0 到 1，一次性流程，产出 docs/ 目录的初始文档
- **product-iteration**：从 1 到 N，反复使用，更新 docs/ 目录的现有文档
- 迭代时会读取并更新 product-builder 产出的 `docs/prd.md`、`docs/tasks.md` 等文件

---

## 总览

4 个阶段，每轮迭代走一遍：

| 阶段 | 做什么 | 产出 |
|------|--------|------|
| 1. 收集信号 | 整理数据、反馈、观察 | `docs/iterations/vN-signals.md` |
| 2. 评估决策 | 筛选优先级，决定做什么 | `docs/iterations/vN-decision.md` |
| 3. 定义范围 | 明确这次迭代改什么、不改什么 | 更新 `docs/prd.md` |
| 4. 拆分执行 | 拆成开发任务 | 更新 `docs/tasks.md` |

每次迭代的记录存在 `docs/iterations/` 目录，形成可追溯的决策历史。

---

## 核心对话规则

与 product-builder 一致：
1. **每次只问 1 个核心问题**，配合 2-4 个选项让用户快速选择
2. 回复简短：2-3 句话 + 1 个问题 + 选项，不要长篇大论
3. 每 2-3 轮简短复述当前理解，确认没有偏差
4. 用数据和事实引导，不凭感觉
5. 语气自然亲切，像一个聪明的朋友在咖啡馆聊天

**迭代特有规则：**
6. 始终先问"有什么数据/证据支持这个判断？"，抵制拍脑袋决策
7. 每个迭代决策必须关联到之前定义的成功指标
8. 敢于建议"不做"——不是每个反馈都值得响应
9. 遇到用户说"顺便也加个 XXX 吧"时，追问："这个和本次迭代目标有关吗？"

## 阶段 1：收集信号

**目标：** 搞清楚现在发生了什么，哪些信号值得关注。

**开场：** 先读取 `docs/product-definition.md` 了解产品背景和成功指标，然后开始对话。

**开场话术示例：**
> "上次我们定义的成功指标是 [从文档读取]。现在情况怎么样——这些指标达到预期了吗？还是某些方面和你想的不太一样？"

**对话流程：**

1. 当前状态（1-2轮）
   - 产品上线多久了？
   - 之前定义的成功指标现在是什么数字？
   - 和预期比，哪些达标了，哪些没达标？

2. 定量信号（2-3轮）
   - 关键数据指标是什么？（用户数、留存、转化、使用频率等）
   - **商业化信号：** 有付费用户了吗？转化率多少？付费用户和免费用户的行为有什么不同？
   - 哪个指标最让你担心？
   - 数据有没有让你意外的地方？（比预期好或差）

3. 定性信号（2-3轮）
   - 用户反馈说了什么？（好的和坏的）
   - 用户在哪一步流失最多？**是在付费墙流失还是在使用中流失？这两个问题的解法完全不同。**
   - 你自己用产品时最大的不爽是什么？
   - 如果用户反馈"不好用"但说不清哪里不好，用 `ux-principles` 的 7 条原则逐个对照定位具体问题

4. 外部信号（1-2轮）
   - 竞品有什么新动作？
   - 市场/行业有什么变化？
   - 有没有新的技术或工具可以利用？

5. 增长漏斗（1轮，用选项快速定位瓶颈）
   - "你的增长瓶颈在漏斗的哪一环？" 给出选项：
     - 获客（用户根本不知道你的产品存在）
     - 激活（用户来了但没体验到核心价值就走了）
     - 留存（用了一次但不回来）
     - 转化（用了但不付费）
     - 推荐（用了也付费了但不推荐给别人）
   - 这一轮的答案会直接影响阶段 2 的优先级排序

**产出文件模板 `docs/iterations/vN-signals.md`：**

```markdown
# 迭代信号收集 - V[N]

**日期：** [日期]
**上次迭代：** V[N-1]（[简述上次做了什么]）

## 成功指标回顾
| 指标 | 目标值 | 当前值 | 状态 |
|------|--------|--------|------|
| [指标1] | [目标] | [实际] | ✅/⚠️/❌ |
| [指标2] | [目标] | [实际] | ✅/⚠️/❌ |

## 定量信号
- [数据发现1]
- [数据发现2]
- 最值得关注的异常：[描述]

## 定性信号（用户反馈）
### 正面反馈
- [反馈1]
- [反馈2]

### 负面反馈 / 痛点
- [反馈1] — 出现频率：[高/中/低]
- [反馈2] — 出现频率：[高/中/低]

### 功能请求
- [请求1] — 请求人数：[N]
- [请求2] — 请求人数：[N]

## 增长漏斗瓶颈
[获客/激活/留存/转化/推荐]

## 自身观察
- [你使用产品时发现的问题]

## 外部信号
- 竞品动态：[描述]
- 市场变化：[描述]
```

## 阶段 2：评估决策

**目标：** 从一堆信号中筛选出这次迭代最该做的事。

**对话流程：**

1. 信号分类（1-2轮）
   - 复述收集到的信号，问用户："我把这些分成三类——修复、优化、新增。你看对不对？"
   - "这里面有没有'止血'级别的——不做的话用户马上流失或者系统会出大问题？"

2. 影响评估（2-3轮）— 对每个候选项逐个问，用选项收敛：
   - "如果做了 [候选项X]，你觉得对核心指标的影响是：大 / 中等 / 小 / 不确定？"
   - "如果不做，最坏情况是什么？用户流失 / 体验下降但能忍 / 没什么影响？"
   - "做这件事大概需要多久？1-2天 / 3-5天 / 一周以上？"
   - 做这件事的成本（时间/复杂度）是多少？

3. 优先级排序（2-3轮）
   - 用 ICE 框架快速打分：
     - **Impact**（影响力 1-5）：做了对核心指标影响多大？**对收入/转化有直接影响吗？**
     - **Confidence**（信心 1-5）：你有多确定它能达到预期效果？
     - **Ease**（容易度 1-5）：做起来有多容易？
   - 按 ICE 总分排序，选出这次迭代的候选清单
   - 问用户：按这个排序，你同意吗？有没有要调整的？

4. 做/不做决策（1-2轮）
   - 确认这次迭代做哪几件事（通常 1-3 件）
   - 明确哪些"先不做，放到下次评估"
   - 问关键问题：如果这次只能做 1 件事，你选哪个？
   - **商业化追问：** "这次迭代的改动会影响收费模式或定价吗？免费和付费的边界需要调整吗？"

**产出文件模板 `docs/iterations/vN-decision.md`：**

```markdown
# 迭代决策 - V[N]

**日期：** [日期]

## 候选清单（ICE 评分）
| # | 事项 | 类型 | Impact | Confidence | Ease | ICE | 决策 |
|---|------|------|--------|-----------|------|-----|------|
| 1 | [事项] | 修复/优化/新增 | [1-5] | [1-5] | [1-5] | [总分] | ✅ 本次做 |
| 2 | [事项] | ... | ... | ... | ... | ... | ✅ 本次做 |
| 3 | [事项] | ... | ... | ... | ... | ... | ⏳ 下次评估 |
| 4 | [事项] | ... | ... | ... | ... | ... | ❌ 不做 |

## 本次迭代目标
用一句话概括：这次迭代要解决什么问题？

## 做的事
1. [事项1]：[为什么做？预期效果是什么？]
2. [事项2]：[为什么做？预期效果是什么？]

## 明确不做的事
- [事项3]：[为什么不做？什么时候重新评估？]
- [事项4]：[为什么不做？]

## 本次迭代的成功指标
| 指标 | 当前值 | 目标值 | 衡量方式 |
|------|--------|--------|----------|
| [指标] | [现在] | [期望] | [怎么量] |

## 预计工期
[N 天/周]
```

## 阶段 3：定义迭代范围

**目标：** 把迭代决策细化成具体的需求变更，更新 PRD。

**对话流程：**

1. 逐项细化（每项 2-3轮）— 对本次迭代要做的每件事：
   - 具体改什么？（哪个页面、哪个流程、哪个数据）
   - 改完之后用户体验是什么样的？
   - 有哪些边界情况要考虑？
   - 会影响现有功能吗？有没有回归风险？

2. 范围确认（1轮）
   - 复述完整的变更清单
   - 确认没有遗漏和越界

**操作：** 读取现有 `docs/prd.md`，在对应功能章节下标注变更：

```markdown
### 功能1：[名称]
- **描述：** [原有描述]

> **V[N] 变更：**
> - [改了什么]
> - [新增了什么]
> - [删除了什么]
```

**原则：**
- 在现有 PRD 上增量修改，不要重写整个文档
- 变更必须标注版本号（VN），方便追溯
- 如果是新功能，在 PRD 末尾新增章节
- 更新 In-Scope / Out-of-Scope 部分

## 阶段 4：拆分执行

**目标：** 把这次迭代的变更拆成开发任务。

**对话流程：**

任务拆分方式复用 `product-builder` 阶段 4 的垂直切片方法和任务模板，但迭代有两个额外关注点：

1. 回归风险评估（1-2轮）
   - 这次改动会影响哪些现有功能？（用 `change-guard` 的影响链做分析）
   - 需要补充哪些测试来确保不回归？

2. 数据迁移（0-1轮）
   - 数据模型有变化吗？
   - 需要迁移脚本吗？
   - 老数据怎么处理？

**操作：** 在 `docs/tasks.md` 中新增迭代章节：

```markdown
---

## 迭代 V[N]：[一句话目标]

**日期：** [日期]
**决策文档：** docs/iterations/vN-decision.md

### Task N.1：[任务名称]
- **目标：** [做什么]
- **变更范围：** [影响哪些文件/模块]
- **回归风险：** [可能影响的现有功能]
- **验收标准（happy path）：**
  - Given [前置条件]，When [操作]，Then [期望结果]
- **验收标准（failure path）：**
  - Given [失败条件]，When [操作]，Then [错误处理]
- **回归测试：** [需要额外跑哪些测试]

### Task N.2：[任务名称]
- ...
```

**执行规范：** 见 `architecture-principles` 工作流 W2。额外要求：
- 每个 Task 开始前用 `change-guard` 做影响分析（迭代改动比新建更容易漏）
- 每个 Task 完成后跑**全量**测试（不只是相关测试）确保不回归
- 所有 Task 完成后：用 `change-guard` 工作流 4 做一次文档同步，更新成功指标
- 如果安装了 claude-workflow 插件：可以用 `/cw-dispatch` 自动派发可并行的 Task，用 `/cw-worktree` 管理并行分支（详见 `product-builder` 任务模板中的"自动化执行"章节）

## 迭代节奏建议

在对话中根据项目阶段建议合适的节奏：

| 阶段 | 建议迭代周期 | 说明 |
|------|-------------|------|
| MVP 刚上线 | 每 1-2 周 | 快速响应，验证核心假设 |
| 初步产品-市场契合 | 每 2-4 周 | 有节奏地改进，不过度反应 |
| 稳定增长期 | 每 4-8 周 | 更大的功能迭代，更多数据积累 |

**每次迭代的 4 个阶段不是都要完整走一遍：**
- 紧急 bug 修复：跳过阶段 1-2，直接定义范围 → 拆任务
- 小优化：阶段 1-2 可以 2-3 轮快速过
- 大功能迭代：完整走完 4 个阶段

## 反模式检测

在对话中主动识别并提醒用户避免这些常见错误：

### 🚫 功能堆砌
**信号：** 用户说"顺便也加个 xxx 吧"
**应对：** "这个功能和本次迭代目标有关吗？如果不直接相关，建议记录到下次评估，保持这次迭代的聚焦。"

### 🚫 数据缺失决策
**信号：** 用户说"我觉得应该..."但没有数据支持
**应对：** "这个判断有什么数据或用户反馈支持吗？如果还没有数据，要不要先加个埋点，下次迭代时用数据来决定？"

### 🚫 过度反应个别反馈
**信号：** 用户因为 1-2 个用户的反馈就要大改
**应对：** "这个反馈有多少用户提到过？如果只是个别情况，可能不值得花大成本改。可以先观察是否有更多类似反馈。"

### 🚫 忽略回归风险
**信号：** 用户想改核心流程但没考虑对现有功能的影响
**应对：** "这个改动会影响 [现有功能X]，我们需要先确认现有测试覆盖了这部分，并且加上新的回归测试。"

### 🚫 永远不砍功能
**信号：** 产品越来越臃肿，但从来不删功能
**应对：** "有没有哪些功能使用率很低？有时候砍掉没人用的功能比加新功能更能改善体验。"

### 🚫 追新技术当产品方向
**信号：** 用户说"现在 AI/XX 很火，我们也加一个吧"
**应对：** "新技术应该增强你产品的核心能力，而不是另起一个方向。问自己：这个技术能让用户在你的核心场景中做得更好吗？还是只是因为别人在做所以你也想做？"

## 文件结构

迭代过程中维护的文件：

```
docs/
├── product-definition.md    # 产品定义（一般不改，除非大转向）
├── prd.md                   # PRD（增量更新，标注版本号）
├── tech-stack.md            # 技术方案（偶尔更新）
├── tasks.md                 # 任务清单（每次迭代追加新章节）
└── iterations/              # 迭代决策历史
    ├── v1-signals.md
    ├── v1-decision.md
    ├── v2-signals.md
    ├── v2-decision.md
    └── ...
```

## 快速入口

| 用户说的话 | 从哪里开始 |
|-----------|-----------|
| "该迭代了" / "下个版本做什么" | 阶段 1（收集信号） |
| "用户反馈了 xxx" / "数据不好" | 阶段 1（从具体信号切入） |
| "要不要加这个功能" | 阶段 2（评估这个具体决策） |
| "这次迭代做 xxx" | 阶段 3（已有决策，直接定义范围） |
| "帮我拆一下这次迭代的任务" | 阶段 4（已有范围，直接拆任务） |
| "紧急 bug 要修" | 跳过 1-2，直接阶段 3→4 |

## 与其他 Skill 的协作

- **product-builder**：提供初始的 docs/ 文档，product-iteration 在此基础上增量更新
- **architecture-principles**：迭代时如涉及架构变更，参考其中的设计原则，特别是回归风险评估时的关注点分离原则
- **change-guard**：迭代的代码修改每次走影响分析→完整性检查，迭代结束后运行文档同步
- **ux-principles**：用户反馈"不好用"时用 7 条原则定位具体问题
