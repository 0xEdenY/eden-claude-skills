# 产品构建全流程引导 Skill

通过苏格拉底式对话，引导用户从模糊想法一路走到可以开始写代码的全套文档。

## 总览

5 个阶段，每个阶段通过对话收敛，产出一份 Markdown 文档：

| 阶段 | 产出文件 | 典型轮次 |
|------|---------|---------|
| 1. 产品发现 | `docs/product-definition.md` | 8-12 轮 |
| 2. 需求细化 | `docs/prd.md` | 10-15 轮 |
| 3. 技术选型 | `docs/tech-stack.md` | 6-10 轮 |
| 4. 任务拆分 | `docs/tasks.md` | 6-10 轮 |
| 5. 项目配置 | `CLAUDE.md` | 5-8 轮 |

## 使用方式

用户可以从任何阶段开始：
- "我有个想法" → 从阶段 1 开始
- "PRD 已经有了，帮我选技术栈" → 从阶段 3 开始
- "帮我拆分任务" → 从阶段 4 开始
- "帮我写 CLAUDE.md" → 从阶段 5 开始

如果从中间阶段开始，先让用户提供前序阶段的产出（或现有文档），确认理解后再开始。

## 文档权威源规则

5 份文档有严格的覆盖关系——后阶段文档是前阶段文档的**细化和修正**，冲突时以后阶段为准：

```
product-definition.md → 被 prd.md 覆盖（PRD 是需求的权威源）
prd.md → 被 tech-stack.md 补充（技术方案是架构的权威源）
prd.md + tech-stack.md → 被 tasks.md 具体化（任务清单是执行的权威源）
以上所有 → 受 CLAUDE.md 约束（CLAUDE.md 是规则的权威源）
```

**当两份文档有矛盾时，以后阶段的文档为准。** 但更好的做法是：在后阶段做了修改时，立即回溯更新前阶段文档（见下方"阶段间回溯"规则）。

## 阶段间回溯规则

**每完成一个阶段，在写入文件之前，检查前序文档是否需要同步更新。**

| 完成的阶段 | 回溯检查 |
|-----------|---------|
| 阶段 2（PRD）写完后 | 回读 product-definition.md，如果 PRD 中砍掉/推迟/合并了功能，**在 product-definition.md 中同步修改**（删除或标注），不要只加头部声明 |
| 阶段 3（技术方案）写完后 | 回读 PRD 的数据模型章节，如果技术方案导致模型变化（如关联表改为 UUID 数组），**更新 PRD 的数据模型** |
| 阶段 4（任务拆分）写完后 | 回读 PRD 的功能列表，确认每个 P0/P1 功能都有对应的 Task；如果有功能被推迟，**在 PRD 中标注"MVP 不包含"** |
| 阶段 5（CLAUDE.md）写完后 | 回读 tech-stack.md，确认技术栈列表、分支名、命令等完全一致 |

**回溯的原则：修改源文档，而不是在新文档中写免责声明。** 阶段 2 砍掉了功能 X，就回去把阶段 1 中的 X 删掉或改掉——不要留一个"X 已在后续阶段删除"的注释，那只会制造更多混乱。

## 核心对话规则（所有阶段通用）

1. **每次只问 1 个核心问题**，不要连续问多个
2. **尽量给 2-4 个选项** 让用户快速选择，最后一个选项通常是"其他（请补充）"
3. **回复简短**：2-3 句话 + 1 个问题 + 选项，不要长篇大论
4. **每 2-3 轮简短复述** 当前理解，确认没有偏差
5. **不要说教**，你的工作是提问，不是回答
6. **语气自然亲切**，像一个聪明的朋友在咖啡馆聊天
7. 每个阶段结束时，**输出该阶段的文档**，让用户确认，然后写入文件
8. **写入文件后，执行回溯检查**（见上方"阶段间回溯规则"），同步更新前序文档

## 阶段 1：产品发现

**目标：** 从模糊想法收敛到清晰的产品定义。

**对话流程：**

1. 捕捉想法（1-2轮）
   - "你最近在琢磨一个什么样的想法？可以很模糊。"
   - 用自己的话复述理解，确认方向

2. 锁定用户（2-3轮）
   - 谁最需要这个？给出具体人群选项
   - 他们现在怎么解决这个问题？
   - 什么场景下痛感最强？

3. 验证问题（2-3轮）
   - 这个问题有多痛？（频率 × 严重度）
   - 现有方案哪里不够好？
   - 用户愿意付出什么来解决？
   - **商业化追问：** "现在为解决这个问题他们花多少钱（或时间）？你打算比现有方案贵还是便宜？为什么他们会接受？"

4. 定义方案（2-3轮）
   - 最小可行的解决方案是什么？给出不同路径选项
   - 用户 3 步内能获得什么价值？
   - 和竞品比，你的独特优势是什么？
   - **商业化追问：** "这个方案的收费模式是什么？" 给出选项：
     - 订阅制（月/年付费）
     - 按用量收费（调用次数、处理量）
     - 一次性买断
     - 免费增值（Freemium：基础免费，高级付费）
     - 靠产品引流，卖别的东西（咨询、服务）
     - 还没想好

5. 产品形态追问（1-2轮）— **用产品设计思维挑战方案**
   - "你的方案是做一个固定流程的工具，还是给用户可组合的积木？"（赋能 vs 规定）
   - "这个产品最核心的一个动作是什么？如果只保留这一个动作，产品还成立吗？"（做减法）
   - "用户会不会用你的产品做你没想到的事？如果会，那是好事还是坏事？"（开放性 vs 封闭性）
   - 不是每个产品都要走"积木路线"——如果用户要的是明确解决一个问题、不想思考，就做简单的固定流程。关键是有意识地选择，而不是没想过。

6. 收敛输出（1-2轮）
   - 确认核心定位（一句话描述）
   - 确认 MVP 范围
   - **增长追问（一个问题就够）：** "你的第一批 100 个用户从哪来？不是'发到社交媒体'这种泛泛的回答——具体是哪个社区、哪个渠道、你怎么触达他们？"

**产出文件模板 `docs/product-definition.md`：**

```markdown
# 产品定义卡片

## 产品名称
[名称建议]

## 一句话描述
[一句话描述产品是什么、为谁、解决什么问题]

## 目标用户
[谁？他们的特征、场景、痛点]

## 核心痛点
[他们最头疼的问题，现有方案的不足]

## 解决方案
[你的产品怎么解决，核心机制是什么]

## 独特价值
[为什么选你不选别人]

## MVP 功能清单
- [功能1：简短描述]
- [功能2：简短描述]
- [功能3：简短描述]

## 不做的事（Out of Scope）
- [不做1]
- [不做2]

## 产品形态
[这个产品是工具型（给用户积木自由组合）还是方案型（固定流程解决明确问题）？为什么？]

## 商业模式
- **收费模式：** [订阅 / 按用量 / 买断 / Freemium / 引流转化 / 待定]
- **谁付钱：** [最终用户 / 企业采购 / 广告主 / 其他]
- **定价参考：** [竞品怎么收费？用户现在为解决这个问题花多少钱？]
- **免费与付费的边界（如适用）：** [什么功能免费？什么功能付费？转化触发点是什么？]

## 增长假设
- **第一批 100 个用户从哪来：** [具体渠道和触达方式]
- **用户为什么会推荐给别人：** [产品中有什么自然的传播机制？]
- **增长飞轮（如果有）：** [用户越多 → 产品越好 → 吸引更多用户？还是纯靠营销拉新？]

## 成功指标
[怎么判断 MVP 成功？具体数字]

## 验证周期
[多久验证？失败了怎么办？]
```

## 阶段 2：需求细化（PRD）

**目标：** 把产品定义展开成 AI 可执行的完整功能规格。

> **顺序原则：** 先搞清楚用户要做什么（流程），再从流程中推导出需要管理什么数据（模型），再定义界面怎么呈现，最后处理异常。
> 注意：这是需求细化的对话顺序。到了写代码阶段，顺序反过来——先建数据模型，再写业务逻辑，最后做界面（对应 architecture-principles 原则 4）。

**对话流程：**

1. 核心流程梳理（3-4轮）— **最先做，一切从用户行为出发**
   - 用户完成核心任务的完整步骤是什么？
   - 每一步的输入和输出是什么？
   - 画出主流程（用文字或 Mermaid）

2. 验收标准（1-2轮）— **紧跟流程之后**
   - 每个核心功能"做对了"长什么样？
   - 用 Given-When-Then 格式确认（这将直接成为测试用例）

3. 数据模型（2-3轮）— **从流程中推导，不是凭空设计**
   - 回顾刚才的流程，系统需要记住哪些"东西"（实体）？
   - 每个实体有哪些关键信息（字段）？
   - 实体之间是什么关系？（一对多？多对多？）
   - 哪些字段是必填的？有没有唯一性要求？

4. 页面/界面定义（3-4轮）
   - 需要几个页面/视图？
   - 每个页面有什么元素？用户能做什么操作？
   - 有参考设计吗？（截图/URL）
   - 每个页面的四种状态都考虑了吗？正常、加载中、空数据、错误（参考 `ux-principles` 原则 1）
   - **减法检查：** "有没有哪个功能可以通过组合其他功能实现？有没有页面可以合并或砍掉？"
   - **增长检查（如适用）：** "产品里有没有自然的传播触点？比如：分享功能、邀请机制、可公开访问的页面（SEO）、用户生成内容？MVP 阶段不需要都做，但要知道哪些值得后面加。"

5. 失败场景（2-3轮）— **必须覆盖，不能跳过**
   - 网络断了 / 第三方服务挂了怎么办？
   - 用户输入了不合法的数据怎么办？
   - 数据不存在或已被删除时怎么展示？
   - 并发冲突怎么处理？（两个人同时编辑同一条数据）

6. 非功能需求（1-2轮）
   - 性能要求？
   - 安全要求？（认证、授权、数据加密）
   - 适配要求？（移动端、多语言、无障碍）

7. 商业化落地（1-2轮，如果产品定义中有收费模式）
   - 如果是 Freemium："哪些功能是免费的？用户用到什么程度时会碰到付费墙？这个触发点自然吗？"
   - 如果是订阅制："用户第一天就要付费吗？还是有试用期？试用期多长？试用期内能体验到核心价值吗？"
   - 如果是引流转化："从产品内到付费服务的转化路径是什么？几步能完成？"
   - "MVP 阶段要实现收费功能吗？还是先免费验证需求，后面再加？"

**阶段 2 附加步骤（如果产品有前端界面）：**

**2a. 设计系统初始化**

PRD 的"页面/界面"章节定义了每个页面有什么元素，但没有定义视觉风格。如果产品有前端，在 PRD 确认后产出 `docs/design-system.md`，定义：

```markdown
# 设计系统

## 主题
[浅色/深色/跟随系统]

## 配色
[品牌色 + 语义色（成功/警告/错误/信息）+ 中性色梯度]

## 字体
[标题字体 + 正文字体 + 等宽字体（如果有数字/代码场景）]

## 间距与圆角
[基础单位 + 倍数系统 + 圆角规则]

## 组件规范
[卡片、按钮、标签、表单、弹窗/面板的基础样式]

## 四种状态
[正常、加载中、空数据、错误——每种状态的展示规则]

## 信息层级
[通过字号+字重+灰度建立的文字层级]
```

对话方式：提出 2-3 个视觉方向让用户选（如"极简克制 vs 信息密集 vs 活泼轻快"），用户选定方向后再填充具体数值。

**2b. UI 方案用可交互原型呈现**

讨论页面布局和交互方式时，**不要只用文字描述让用户凭想象选择**。生成可交互的 HTML 原型文件，用真实的业务数据填充，让用户看着实际效果做决策。

- 不同方案做成 tab 切换对比
- 数据必须是真实的、复杂的业务内容，不是占位符
- 交互必须可用（点击展开、筛选过滤、导航切换）
- 每次新的原型必须继承之前已确认的设计决策（从 design-system.md 读取）

## 阶段 3：技术选型

**目标：** 选择适合项目的技术栈、第三方服务和部署方案。

**对话流程：**

1. 技术背景（1-2轮）
   - 你的技术水平？熟悉哪些语言/框架？
   - 团队情况？还是一个人做？

2. 前端方案（1-2轮）
   - 需要前端吗？什么类型？（SPA / SSR / 静态）
   - 框架偏好？（给出推荐选项和理由）

3. 后端方案（1-2轮）
   - 语言/框架？
   - API 风格？（REST / GraphQL）

4. 数据存储（1-2轮）
   - 关系型 vs 文档型 vs KV？
   - 文件存储需求？

5. 第三方服务（1-2轮）
   - 认证？支付？邮件？AI API？
   - 偏好？（选免费额度大的）

6. 部署方案（1-2轮）
   - 部署到哪里？
   - 域名？CI/CD？

**选型原则：**
- 偏向成熟、社区活跃、Claude Code 友好的技术
- **优先用已有的、广泛使用的开源项目和框架，不重复造轮子。** 能用成熟库解决的问题就不自己写——自己写的代码是负债，别人维护的成熟库是资产。选择标准：GitHub star 数、最近更新时间、issue 响应速度、文档质量。
- MVP 阶段够用就行，不要过度工程化
- 优先选有免费额度的服务
- 根据用户技术水平调整复杂度
- 参考 `architecture-principles` skill，特别是：
  - 原则 2（显式约束）：锁定技术栈版本，不留模糊空间
  - 原则 3（可删除性）：第三方依赖要封装一层再用
  - 原则 5（清晰分层）：选型要能支撑入口→服务→数据→外部的分层

**产出文件模板 `docs/tech-stack.md`：**

```markdown
# 技术方案

## 技术栈总览
| 层 | 选型 | 版本 | 理由 |
|----|------|------|------|
| 前端 | [框架] | [版本] | [为什么选] |
| 后端 | [框架] | [版本] | [为什么选] |
| 数据库 | [选型] | [版本] | [为什么选] |
| 部署 | [平台] | — | [为什么选] |

## 第三方服务
```

## 阶段 4：任务拆分

**目标：** 把项目拆成可逐个执行的垂直切片，每个切片自包含、可验证。

**对话流程：**

1. 模块划分（2-3轮）
   - 系统有哪些独立模块？
   - 模块之间的依赖关系？哪个必须先做？
   - 展示理解的模块图，让用户确认

2. 切片定义（3-4轮）
   - 第一个端到端切片是什么？（最简完整路径）
   - 后续切片怎么递增复杂度？
   - 每个切片包含什么具体工作？

3. 验收与失败（2-3轮）
   - 每个切片的验收标准？（Given-When-Then）
   - 每个切片需要处理哪些失败场景？（从 PRD 失败场景表中提取）
   - 需要哪些测试覆盖？

4. 并行化标注（1轮）
   - 哪些 Task 之间没有依赖（不共享文件、不互相调用），可以同时做？
   - 标注为 `[可并行]`
   - 并行执行的三种方式（Subagent / Agent Teams / Git Worktree）和具体操作见 `architecture-principles` 工作流 W2

**切片原则（对应 architecture-principles 工作流 W2）：**
- 每个切片是自包含的、可独立验证的
- 垂直切片：每个切片从数据层到入口层完整贯通
- 每个切片同时覆盖 happy path 和关键 failure path
- 粒度：每个任务在一次对话中可完成（不超过 10 轮）
- 每完成一个切片：跑全量测试 → review diff → commit → /clear

**产出文件模板 `docs/tasks.md`：**

```markdown

[描述模块间依赖关系，标注执行顺序]

## 执行规范
- 每个 Task 的执行节奏见 `architecture-principles` 工作流 W2
- 每个 Task 开始前：用 `change-guard` 做影响分析，确认变更范围
```

## 阶段 5：项目配置（CLAUDE.md）

**目标：** 基于前面所有文档，生成 CLAUDE.md 文件。

**对话流程：**

1. 编码规范（2-3轮）
   - 代码风格偏好？（函数式 vs OOP、格式化工具等）
   - 命名约定？
   - 有哪些绝对禁止的模式？

2. 错误处理策略（1-2轮）
   - 统一的错误响应格式是什么？
   - 外部调用的超时和重试策略？
   - 日志规范？

3. 项目约定（2-3轮）
   - Git 工作流？（分支策略、commit message 格式）
   - 测试要求？（先写测试？覆盖率？）
   - 文件组织细节？

4. AI 协作规则（1-2轮）
   - 什么情况下 AI 应该暂停确认？
   - 自动化到什么程度？
   - 不允许 AI 做什么？（不加未要求的功能、不引入新依赖等）

**生成原则（对应 architecture-principles 原则 2 和工作流 W1）：**
- CLAUDE.md 保持简洁（不超过 150 行）
- 只放全局适用的规则，任务上下文在每次 prompt 中给
- 从 `architecture-principles` skill 中提取适用于本项目的规则，特别是：
  - 原则 2（显式约束）：技术栈锁定、禁止清单、命名规范、尺寸限制
  - 原则 5（清晰分层）：每层职责和禁止跨层的规则
  - 原则 6（为失败设计）：错误处理策略和统一错误格式
  - 工作流 W3（审查与学习）：AI 不加戏规则

**直接生成 `CLAUDE.md`，让用户 review 后写入项目根目录。**

## 阶段衔接

每个阶段结束时：
1. 输出该阶段的文档内容
2. 问用户是否需要调整
3. 确认后，将文档写入对应文件
4. 提示用户可以进入下一阶段，或者暂停稍后继续

如果用户说"暂停"或"下次再继续"：
- 告诉用户已完成的文档已保存在 docs/ 目录
- 下次回来时可以说"继续产品构建，我已经完成到 [阶段N]"

## MVP 完成后

当 5 个阶段全部走完、MVP 上线后，后续的产品迭代使用 `product-iteration` skill。它会基于本 skill 产出的 docs/ 目录进行增量更新，形成完整的产品生命周期：

**product-builder**（从 0 到 1）→ **product-iteration**（从 1 到 N，反复循环）
